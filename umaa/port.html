FastDDS SHM — Boost Segment 구조 완전 해부

SharedMemGlobal · SharedMemManager · MultiProducerConsumerRingBuffer  |  boost::interprocess 기반

1. 두 종류의 SHM 세그먼트

FastDDS SHM 트랜스포트는 역할이 전혀 다른 두 종류의 shared_memory 세그먼트를 사용합니다.

포트 세그먼트
신호 전달 전용 — 링버퍼 컨테이너
이름: {domain}_port{port_id}
예) fastdds_port7412

소유자: 포트에 접근하는 최초 프로세스가 생성.
ref_counter가 0이 되면 마지막 소유자가 직접 삭제.

내용: PortNode + Cell 배열 + RingBuffer::Node
실제 메시지 데이터는 없음 — 참조(descriptor)만 저장
데이터 세그먼트
데이터 보관 전용 — Publisher 소유
이름: {domain}_{uuid}
예) fastdds_a3f2b8c1d4e5...

소유자: Publisher 프로세스가 생성 및 소유.
소유자 종료 시 RobustExclusiveLock(_el) 해제 → Watchdog이 감지 후 정리.

내용: BufferNode 배열 + 직렬화된 RTPS 메시지 원본
zero-copy의 핵심 — Subscriber는 여기를 직접 읽음
왜 분리하나?  포트 세그먼트는 여러 프로세스가 공유하는 신호 채널이고, 데이터 세그먼트는 Publisher 하나가 소유하는 메모리 풀입니다. 수명·소유권·크기 요건이 완전히 다르기 때문에 분리합니다.

세그먼트별 락 파일 규칙

파일명종류용도
{domain}_port{id}_mutexboost named_mutex포트 세그먼트 생성/삭제 경쟁 방지
{domain}_port{id}_elRobustExclusiveLockReadExclusive 모드 단독 리스너 잠금
{domain}_port{id}_slRobustSharedLockReadShared 모드 공유 리스너 잠금
{domain}_{uuid}_elRobustExclusiveLock데이터 세그먼트 생존 표시 (프로세스 크래시 감지용)

2. boost::interprocess 동작 원리

세그먼트 생성 / 열기

// 생성 (최초 프로세스)
new SharedMemSegment(
    boost::interprocess::create_only,
    "fastdds_port7412",
    segment_size            // 바이트 단위 총 크기
);

// 열기 (다른 프로세스)
new SharedMemSegment(
    boost::interprocess::open_only,
    "fastdds_port7412"
);

두 프로세스는 서로 다른 가상 주소에 같은 물리 메모리를 매핑합니다. OS가 페이지 테이블로 연결.

객체 배치 — construct<T>()

// 이름 있는 객체 (다른 프로세스가 이름으로 조회 가능)
PortNode* node = segment.construct<PortNode>(
    "port_node_abi5"   // 이름
)();                   // 생성자 인수

// 익명 배열 (오프셋으로만 접근)
Cell* cells = segment.construct<Cell>(
    boost::interprocess::anonymous_instance
)[max_buffer_descriptors]();

// 이름으로 찾기 (다른 프로세스에서)
PortNode* node = segment.find<PortNode>(
    "port_node_abi5"
).first;

원시 메모리 할당 — allocate()

// 데이터 세그먼트에서 메시지용 메모리 할당
void* data = segment.allocate(size);

// 세그먼트 내부 힙(free-list)에서 꺼냄
// segment.allocate()는 Thread-safe (내부 mutex)

// 사용 완료 후 반환
segment.deallocate(data);

크로스 프로세스 포인터 — 오프셋 변환

// 포인터 → 오프셋 (SHM에 저장할 때)
Offset off = segment.get_offset_from_address(ptr);
// = ptr - segment_base_address (이 프로세스 기준)

// 오프셋 → 포인터 (SHM에서 읽을 때)
void* ptr = segment.get_address_from_offset(off);
// = other_segment_base_address + off (다른 프로세스 기준)
SHM에 절대 포인터를 저장하면 안 됩니다. 각 프로세스의 매핑 베이스 주소가 다르기 때문에 다른 프로세스에서 역참조하면 segfault. 반드시 오프셋으로 저장해야 합니다.

3. 포트 세그먼트 내부 레이아웃

이름: {domain}_port{port_id}  ·  크기: sizeof(PortNode) + sizeof(Cell) × max_buffer_descriptors + 512

boost managed_shared_memory  ·  포트 세그먼트 fastdds_port7412
+0x000 boost interprocess 내부 헤더 — 힙 메타데이터, named-object 인덱스 ~512 B boost
+0x200 PortNode  ·  construct<PortNode>("port_node_abi5")
ref_counter · port_id · uuid · empty_cv · empty_cv_mutex · listeners_status[1024] · is_port_ok · buffer(offset) · buffer_node(offset) ~33 KB named
+0x8400 Cell[max_buffer_descriptors]  ·  construct<Cell>(anonymous_instance)[N]()
각 Cell = { atomic<uint32_t> ref_counter_ , BufferDescriptor data_ } N × 20 B anon
+0x8400
+N×20 RingBuffer::Node  ·  construct<Node>(anonymous_instance)()
atomic<PtrType> pointer_ { write_p:32 | free_cells:32 } · total_cells_ · registered_listeners_ 16 B anon

Cell 구조 — 링버퍼의 한 칸

struct Cell {
    std::atomic<uint32_t> ref_counter_;   // 아직 pop하지 않은 Listener 수
    T                   data_;            // = BufferDescriptor (실제 링버퍼 내용물)
};

struct BufferDescriptor {              // T = BufferDescriptor
    SharedMemSegment::Id     source_segment_id;    // 데이터 세그먼트 UUID
    SharedMemSegment::Offset buffer_node_offset;   // 세그먼트 내 BufferNode 오프셋
    uint32_t                 validity_id;          // 버퍼 무효화 버전 번호
};
링버퍼 Cell에는 실제 메시지가 없습니다. 오직 "어느 세그먼트의 어느 위치에 메시지가 있는지"를 가리키는 20바이트 참조만 들어갑니다. 이것이 zero-copy의 핵심입니다.

4. 데이터 세그먼트 내부 레이아웃

이름: {domain}_{uuid}  ·  크기: Publisher가 create_segment(total_size, max_allocations)에서 지정한 크기 + 내부 할당 오버헤드

boost managed_shared_memory  ·  데이터 세그먼트 (Publisher 소유) fastdds_a3f2b8c1d4e5f6a7...
+0x000 boost 내부 헤더 — 힙 관리 구조 (free-list allocator) ~512 B boost
+0x200 BufferNode[max_allocations]  ·  construct<BufferNode>(anonymous_instance)[N]()
N개의 슬롯. 각각 atomic<Status> + data_size + data_offset 보유 N × 16 B anon
가변 메시지 데이터 #0  ·  segment.allocate(msg_size)
직렬화된 RTPS 패킷 원본. Publisher가 data() 포인터로 직접 씀 가변 alloc
가변 메시지 데이터 #1, #2, ...
복수의 메시지가 동시에 존재 가능 (아직 모든 Subscriber가 읽지 않은 것들) 가변 alloc
— free space — 힙 여유 공간 — free

BufferNode 필드 상세

struct BufferNode {
    struct Status {
        uint64_t validity_id     : 24;  // 무효화 버전 (BufferDescriptor와 대조)
        uint64_t enqueued_count  : 20;  // 몇 개 포트 큐에 들어가 있는지
        uint64_t processing_count: 20;  // 몇 개 Listener가 처리 중인지
    };
    std::atomic<Status>       status;       // 64-bit CAS로 세 값을 동시 갱신
    uint32_t                  data_size;    // 메시지 바이트 수
    SharedMemSegment::Offset  data_offset;  // 세그먼트 내 메시지 원본 오프셋
};
free_buffers_ 리스트
할당 전 BufferNode 풀
foonathan::memory::list<BufferNode*>
세그먼트 생성 시 N개를 모두 여기에 넣음. alloc_buffer()가 뒤에서 꺼냄 (pop_back).
allocated_buffers_ 리스트
사용 중 BufferNode 추적
foonathan::memory::list<BufferNode*>
alloc_buffer() 후 여기에 추가. recover_buffers()가 순회하며 미참조 버퍼를 free_buffers_로 돌려보냄.

5. 프로세스 간 전체 메모리 맵

세 프로세스(Publisher, Subscriber A, Subscriber B)가 두 종류의 세그먼트를 어떻게 공유하는지 보여줍니다.

6. 오프셋 기반 주소 변환 — 왜 필요한가

문제: 각 프로세스의 매핑 주소가 다름

// 물리 메모리 페이지 = 동일
// 가상 주소 = 프로세스마다 다름

Publisher  : SHM 베이스 = 0x7f3a_0000
Subscriber : SHM 베이스 = 0x7f8b_0000

// Publisher가 SHM에 포인터를 저장하면:
ptr = 0x7f3a_0200  // Publisher 기준 주소
// Subscriber가 역참조하면:
// *0x7f3a_0200 → segfault (자신의 VA 공간에 없음)

해결: 오프셋만 SHM에 저장

// Publisher가 저장할 때:
offset = ptr - 0x7f3a_0000 = 0x200
// SHM에는 offset = 0x200 저장

// Subscriber가 읽을 때:
ptr = 0x7f8b_0000 + 0x200 = 0x7f8b_0200
// → 같은 물리 페이지를 자신의 VA로 접근

// 코드:
Offset off = seg->get_offset_from_address(ptr);
void* local = seg->get_address_from_offset(off);

이 원칙이 코드 전체에 일관되게 적용됩니다:

저장 위치오프셋 필드가리키는 대상
PortNode (포트 세그먼트 내)buffer같은 포트 세그먼트의 Cell 배열 시작
PortNode (포트 세그먼트 내)buffer_node같은 포트 세그먼트의 RingBuffer::Node
BufferNode (데이터 세그먼트 내)data_offset같은 데이터 세그먼트의 메시지 원본
BufferDescriptor (Cell.data_ 내)buffer_node_offset데이터 세그먼트의 BufferNode
BufferDescriptor (Cell.data_ 내)source_segment_id어느 데이터 세그먼트인지 UUID 식별

7. 한 메시지의 전체 여정 (주소 변환 포함)

1
Publisher: 데이터 세그먼트에서 버퍼 할당
Segment::alloc_buffer(msg_size) → free_buffers에서 BufferNode 꺼냄 → segment.allocate(msg_size)로 힙 공간 확보 → buffer_node.data_offset = get_offset_from_address(data_ptr) 저장
2
Publisher: SharedMemBuffer.data()로 메시지 직접 쓰기
data_ = segment.get_address_from_offset(buffer_node.data_offset) — 이 포인터로 직접 SHM에 RTPS 패킷 직렬화. memcpy 없음.
3
Publisher: Port에 BufferDescriptor push
Port::try_push(buffer) → inc_enqueued_count → Global::Port::try_push(BufferDescriptor{segment_uuid, buffer_node_offset, validity_id}) → RingBuffer::push()로 Cell에 기록, ref_counter = registered_listeners → empty_cv.notify
4
Subscriber: empty_cv 대기 → head() 확인
wait_pop()에서 empty_cv.timed_wait() 깨어남 → global_listener_->head()로 Cell 포인터 획득 (자신의 read_p == write_p면 nullptr) → Cell.data() = BufferDescriptor 복사
5
Subscriber: 데이터 세그먼트 찾기 + 매핑
find_segment(descriptor.source_segment_id) → ids_segments_ 캐시 확인 → 없으면 open_only로 매핑 → SegmentWrapper::WatchTask에 등록 → SharedMemSegment 반환
6
Subscriber: 오프셋으로 BufferNode 접근 → validity 검증
BufferNode* node = segment.get_address_from_offset(descriptor.buffer_node_offset) → node.is_valid(descriptor.validity_id) 검사 → 불일치시 Publisher가 이미 invalidate → 버퍼 버리고 루프
7
Subscriber: SharedMemBuffer 생성 → 메시지 직접 읽기
SharedMemBuffer(segment, segment_id, buffer_node, validity_id) 생성 → data_ = segment.get_address_from_offset(buffer_node.data_offset) → 앱에서 buffer->data()로 포인터 접근. 복사 없음.
8
Subscriber: pop() → 카운터 갱신 → 버퍼 자동 반환
Cell.ref_counter_.fetch_sub(1) → 0이 되면 free_cells++(CAS) → SharedMemBuffer 소멸 시 dec_processing_count(validity_id) → enqueued==0 && processing==0이면 Segment가 다음 recover_buffers()에서 segment.deallocate(data_ptr)

8. 세그먼트 생명주기 요약

포트 세그먼트 생명주기
생성: open_or_create — 존재하면 열고, 없으면 생성. named_mutex로 경쟁 방지.

공유: ref_counter 증감. 열 때 +1, 닫을 때 -1.

소멸: ref_counter가 0이 되면 마지막 소유자가 SharedMemSegment::remove() 호출.

비정상 종료: WatchTask가 리스너 동결을 감지 → is_port_ok=false → 다음 접근 시 예외 → regenerate_port()로 새 세그먼트 재생성.
데이터 세그먼트 생명주기
생성: Publisher가 create_only로 생성. UUID 충돌 시 최대 16회 재시도.

생존 표시: segment_name + "_el" 파일에 RobustExclusiveLock. 프로세스가 살아있는 한 유지.

소멸: Publisher 종료 → Segment 소멸자 → remove() + OS가 마지막 매핑 해제 시 물리 메모리 반환.

비정상 종료: SegmentWrapper::WatchTask가 5초마다 _el 잠금 확인 → 잠금 해제되면 close_and_remove().
