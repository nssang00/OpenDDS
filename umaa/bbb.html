Cell·Node·Listener 관계 및 메시지 흐름 전체 구조

실제 데이터는 Cell에 없다  ·  Node/Listener의 위치와 관리 방식  ·  토픽 메시지가 개별 구독자에게 전달되는 과정

핵심 답변: Cell에는 실제 데이터가 없습니다. Cell에는 오직 BufferDescriptor(참조값 20바이트)만 있습니다.
실제 데이터는 Publisher가 소유한 별도 SHM(데이터 세그먼트)에 있고, Subscriber는 descriptor를 받은 뒤 그 세그먼트에서 직접 읽습니다.
RingBuffer::Node는 포트 SHM 안에, Listener C++ 객체는 각 프로세스의 로컬 힙에 존재합니다.

① 모든 객체의 위치 지도 — SHM vs 로컬 힙

② Node와 Listener의 생성·삭제 과정

③ 토픽 메시지 → 개별 구독자 전달 전체 흐름

Publisher Process 1개, Subscriber A/B가 같은 포트에 있는 경우 (같은 참가자, 두 DataReader). 포트는 구독자 참가자별로 하나입니다.

④ 핵심 정리

Cell에는 실제 데이터가 없습니다. Cell.data_ = BufferDescriptor = {UUID, offset, validity_id} — 20바이트짜리 포인터 역할만 합니다. 실제 RTPS 메시지 바이트는 Publisher 소유 데이터 SHM에 있고, Subscriber는 SHM을 직접 매핑해서 읽습니다 (zero-copy).
RingBuffer::Node는 포트 SHM 안에, Listener C++ 객체는 각 프로세스 로컬 힙에 있습니다. Node의 pointer_(write_p+free_cells)와 registered_listeners_는 SHM이므로 모든 프로세스가 공유합니다. Listener의 read_p_는 로컬이므로 구독자마다 독립적입니다.
각 구독자에게 descriptor를 개별로 넣지 않습니다. Cell 하나에 descriptor 하나를 넣고, ref_counter = registered_listeners로 "아직 안 읽은 수"를 세는 방식입니다. 각 구독자는 자신의 read_p_로 같은 Cell을 독립적으로 가리키다가 pop()으로 ref_counter를 1씩 감소시킵니다. 마지막 구독자가 pop하면 Cell 슬롯이 재사용 가능해집니다.
